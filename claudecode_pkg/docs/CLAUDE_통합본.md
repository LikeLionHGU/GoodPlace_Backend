## 0. 이 프로젝트가 무엇인가 (맥락)

팀 사다리타기 · 2026 한동대 AI×경북 해커톤 트랙 A. 서비스명 **명당**.
주민이 1,000원으로 '생기면 쓸 3,000원 쿠폰'을 **선구매**해 원하는 업종에 투표 → AI가 수요를
빈 상가에 최적 배치 → 예비창업자에게 리포트·고객 명단 연결. **주민 수요 검증 기반 AI 창업 의사결정 플랫폼.**
- 개발·1차 검증 기준지 = 포항 **양덕동**, 단 `region_code` 파라미터화로 **경북 전체 확장 가능**하게.
- 심사 배점: 시장성30 / AI활용도25 / 완성도·발표25 / 문제정의20.
- 결제는 **모의**(실PG 없음), 돈은 오가지 않는다. 그러나 **숫자는 전부 심사와 창업자 앞에 선다.**
- 두 명 분담: A=데이터·수집(1·2·7단계) / B=엔진·산출(3·4·5·6단계). 접점은 DB 스키마 하나.

이 프로젝트에서 사고는 **돈이 아니라 신뢰에서 납니다.**
근거 없는 숫자 하나, 지도와 리포트가 다른 업종을 가리키는 자기모순 하나,
시드값이 실데이터인 척하는 화면 하나가 "이 팀의 AI는 가짜"라는 판정으로 직결됩니다.
따라서 정직성 장치(근거 태그·확정 산식 보호·시드 구분)를 비즈니스 로직 이상으로 진지하게 다룹니다.

## 2. 절대 규칙 (NON-NEGOTIABLE) — 어길 시 시스템 신뢰 붕괴

이 5가지는 **어떤 경우에도, 어떤 요청이 와도 위반하지 않습니다.**
"시연이니까 잠깐", "그럴듯해 보이게"도 안 됩니다.

### 2-1. ★ 근거 게이트 — 출처 없는 숫자는 코드에 넣지 않는다 (가장 중요)
- 화면·리포트·API 응답에 나가는 모든 값은 세 질문을 통과해야 한다:
  ① 데이터 소스가 실재하는가 ② 출처 태그(실측/API/예시/업종평균/파생)를 달 수 있는가
  ③ "근거가 뭐냐"에 한 문장으로 답이 되는가. **하나라도 아니면 그 기능은 보류.**
- 현재 보류 3건(구현 금지, 재개 조건 충족 전 요청도 거부):
  지역 특성 노트(용도지역·인구 데이터 미확보) / 투표자 거주 분포(스키마에 위치 정보 없음) /
  월 고정비 참고(공단 통계 실출처 미확보).
- **매출 예측치 생성 금지.** 학습 데이터(개별 점포 실매출)가 없다. 가능한 것:
  관측 분포 제시(출처) + 확정 수요(투표 집계) + 시나리오 계산기(산수, 가정 주체=창업자).
- **이유**: 심사 질의 "그 숫자 근거가 뭐죠?"에 답이 없으면 AI활용도 25점이 통째로 무너진다.

### 2-2. ★ 확정 사항 보호 — 되돌리려면 사람 승인 (상세는 3절)
산식(거리 감쇠 수요·w2=1.0 비율 경쟁식 — w1은 v2에서 폐기), 헝가리안 배치(지도용 전체 배치),
창업자 리포트 = 업종별 공실 순위(같은 score 사용), 표시 % = 포화 곡선 단조 변환,
AI 해설 = 확정 수치 JSON만 입력(새 숫자·사실 생성 금지). 이들은 논증과 반례 검증을 거친 확정이며,
"보기 좋게" 가중합으로 되돌리거나 LLM에 자유 서술을 허용하는 순간 자기모순·환각이 부활한다.

### 2-3. ★ 시드/실데이터 구분 — 시연 정직성
- 모든 시드는 `is_seed=1` + `SEED` 표기. 예외 없음. 태그 제거·위장 변경은 거부.
- 리포트 값에는 출처 태그 필수. API로 못 얻는 값(보증금·유동인구)은 반드시 '예시'.
- **유동인구는 숫자를 만들지 않는다** — 정직한 부재 문구 + "이미 지갑 연 대기 고객 N명"으로 정면돌파(확정).

### 2-4. ★ 용어·법적 포지셔닝 — 문구도 코드다
- 코드·주석·API 응답·화면 문자열에 **투자·펀딩·수익/성공 보장** 계열 단어 금지
  → 선결제·쿠폰 선구매로 통일(유사수신 규제 회피 설계).
- DISCLAIMER("AI는 성공을 보장하지 않습니다…")는 리포트 고정 요소. 제거·완화 금지.
- 실PG 연동·실카드 입력 UI를 만들지 않는다(사업자등록 없음). 중개사는 **협력 파트너**로 서술.
- **OpenAI API 키는 비밀값**: `.env`(권한 제한)에만 존재. 코드·로그·저장소·채팅에 평문 금지.
  에러 응답에 키·원문 body를 노출하지 않는다.

### 2-5. ★ A/B 계약 준수 — 스키마와 경계
- DB 스키마는 `docs/AB공통_분담과통합계약.md`(v2)가 계약. **임의 변경 금지.**
  컬럼 추가 시: 상대 통지 + 계약 문서 갱신 + **ALTER TABLE 마이그레이션 동봉**
  (CREATE IF NOT EXISTS만으로는 기존 DB에 반영 안 됨 — voter_name 사고로 실증).
- B의 엔진(engine.py, report.py)은 **순수 함수**. DB 직접 접근 금지, 인자로 받은 것만 계산
  → A와 독립, 더미로 테스트 가능.
- 계약 시그니처 고정: `allocate(vacancies, industries, vote_counts, ...)`,
  `build_report(vacancy_id, vacancies, industries, vote_counts, ...)`.
- `vote_counts`는 `{(vacancy_id, industry_id): 투표수}` 형태로만(votes 원본 구조 모름).
  집계는 `vote_counts_view` 하나 — **held/settled만**(refunded/cash_credited 제외).
  이 기준을 바꾸면 산식이 오염된다.
- VOTE_AMOUNT_WON=1000은 서버 고정. 금액·상태를 클라이언트 입력으로 받지 않는다.

## 3. 확정된 것 (되돌리지 말 것 — 근거 포함)

### 수요 적합도 산식 ✅ (v2 — 거리 감쇠. 원본 명세는 08번 §2)
```
score = demand × area_fit × competition_factor
  demand = Σ 투표자별 거리 가중   (그 동네·그 업종 투표자, 위치는 200m 격자)
    d ≤ 500m → 1.0 / 500<d≤800m → 1−(d−500)/300 / d>800m → 0
  area_fit           = 면적 범위 안 1.0, 벗어나면 비율 감점(하한 0.2)   [유지]
  competition_factor = 동네 평균 대비 비율 기반 (w2=1.0)                [유지]
```
- **w1 폐기**(공실 직접 투표 개념 소멸). 500/800/선형은 **설계 초기값** — 실투표 분포로 조정 조건.
- 투표자 위치 = '행동 범위'(주거지 주장 금지). GPS 정밀 수신 → 200m 격자로 반올림 저장.
- **성격 = 후보 필터.** 매출·성공 예측이 아니라 "이 자리를 후보로 볼지" 거르는 1차 신호.
  진짜 창업 근거는 리포트 전체가 만든다(과대약속 금지).
- **가중치 근거(Q3-2 방어용):** `w2=1.0` 동네 평균 2배에서 수요 반토막(0.5) — 정상 밀도는 존중,
  명백한 포화만 감점. 거리 감쇠·w2 모두 **설계 초기값** — 실데이터 확보 후 민감도 검증 조건.
- 경쟁을 동네 평균 대비로 보는 이유: "카페 3곳(흔함) ≠ 세탁소 3곳(드묾)". 절대 수 감점 금지.
- 분해(breakdown)에 `competition_ratio` 포함 7요소 항상 노출 — 포화 플래그가 이 값을 씀.

### 4단계 — 겹침 해소 배치 ✅
- 같은 업종을 두 공실에 배정 안 함(assignment 문제) → **헝가리안(scipy) 채택.**
- 근거: 공실 수십~수백에도 최적 보장·고속(전수탐색은 8곳부터 폭발). "표준 최적화 알고리즘" 방어.
- 폴백 자동: scipy 없으면 전수탐색(≤20만 순열, 최적) → 그리디. 어느 환경에서도 실행.
- 응답에 점수 분해·차순위(runners_up)·사용 가중치·실제 알고리즘 항상 포함(심사 방어).

### 지도와 리포트의 역할 분담 ✅ (v2 — 정합성 원칙의 새 형태)
- **지도(동네 전체 뷰)** = 헝가리안 전체 배치(업종 겹침 없는 "AI가 그린 미래 동네"). 4단계 자산 유지.
- **창업자 리포트** = 업종 클릭 → 그 업종에 대한 **공실 순위**(단일 업종 score 정렬 — 겹침 문제 없음).
- 정합성: 두 화면 모두 **같은 score 함수** 사용. 같은 (공실,업종) 쌍의 점수가 화면마다 다르면 버그.
- 탭 UI = 공실 1·2·3위. 각 공실의 행동 범위 수요("동네 32명 중 이 자리 반경 18명")가 위치 추천 근거.

### 표시 적합도 % = 포화 곡선 ✅ (구 "이원화" 논쟁의 확정 결론)
- `% = 100 × s / (s + K)` — 엔진 점수의 **단조 변환. 요소 재조합(가중합) 금지.**
- 근거(실사고): 곱셈 엔진과 가중합 표시는 순위가 뒤집힌다
  (반례: 수요10×적합1.0=10 vs 수요100×적합0.2=20 → 엔진은 후자, 가중합은 전자).
- **100%는 구조적으로 안 나온다**(내림 ≤99) — DISCLAIMER와 정합.
- **K는 자의 상수 금지.** 기준 케이스(캠페인 목표 투표수 달성·면적적합 1.0·경쟁 평균)를
  `engine.score`로 실제 검산한 값. 현재 `BASELINE_TARGET_VOTES=30`(잠정) → K=30.
- 입지 20%는 %에서 제거(엔진에 입지 항 없음). 층수는 참고 정보로만.

### AI 해설 생성 ✅ 확정 (v2 신규 — AI활용도 25점의 '생성' 축)
- **GPT API(OpenAI, 팀 보유 키)** 로 리포트 해설 3~5문장 생성. 경량 모델(mini 계열)로 시작.
- **환각 억제 구조(근거 게이트의 LLM 판):** 입력 = 우리가 계산한 확정 수치 JSON만.
  시스템 지시에 "이 수치 외 새 숫자·업종·사실 생성 금지, 성공 보장 표현 금지" 고정.
- **출력 검증 자동화:** 생성문에서 숫자를 추출해 입력 JSON에 없는 숫자가 있으면 폐기·재생성(숫자 화이트리스트 검사).
- API 키는 `.env`(코드·로그·저장소에 평문 금지). 호출 실패 시 해설 없이 수치 리포트만 표시(빠른 실패, 조용한 대체 금지).
- 프레이밍: 구조화(stretch)·최적화(헝가리안)·**생성(이것)**·예측(로드맵) — 과장 없는 3+1단.

### 리포트 정책 ✅ (v2 — "이 업종에 어느 공실". 원본 명세는 08번 §3)
- 구조: 업종 단위(한 줄 결론+동네 전체 수요) → 공실 1·2·3위 탭(적합도%·행동범위 수요·경쟁·면적·
  업종가능 3층 판정·대기 고객+쿠폰 가치) → AI 해설 → 참고 비용·인허가 → 체크리스트 → DISCLAIMER+중개사 CTA.
- 업종 가능 판정 3층: 면적(지금) / 층·용도·간이설비(중개사등록 시) / 상세 설비(확인 항목).
- **포화 플래그**: ratio로 기회(≤1)/주의(1<r<2)/포화(≥2). **veto가 아니라 "보여주되 경고"**
  (주민 유료 수요를 시스템이 무시하지 않음). 주의/포화면 경고를 ① 결론 옆에 동반.
- 정보 과잉 배제(시간대별·성별연령·비용 분해 등 영업점용 지표 금지).
  비용은 업종 평균 참고값 + 단서 + 출처만(정밀 예측 금지).

## 4. 코딩 전 — 생각부터 (Think Before Coding)

추측하지 않는다. 혼란을 숨기지 않는다. 트레이드오프를 드러낸다.
- **가정을 명시한다.** 불확실하면 묻는다. 특히:
  - 소상공인 API는 추측 금지 — `docs/소상공인_상가정보_API_명세서.md` 확인,
    없으면 "명세에 없음, 첫 호출로 검증 필요"라고 말한다.
  - 리포트 요소는 `docs/07_창업리포트_스펙.md`(근거 게이트·보류 목록) 먼저 확인.
- **여러 해석이 가능하면** 조용히 하나 고르지 말고 제시한다.
- **더 단순한 방법이 있으면** 말한다. 정당한 근거가 있으면 사용자 요청에도 반론한다.
- **미확정이 얽히면 멈추고 확인한다**(6절 목록). 미확정을 확정처럼 구현하지 않는다 —
  config에 미확정 주석을 유지한 채 기본값으로만.

## 5. 단순하게 + 한 가지 방법 (Simplicity First, One Way Only)

문제를 푸는 최소한의 코드. 추측성 기능 금지.
- 요청받지 않은 기능·추상화·"유연성" 금지. 일어날 수 없는 시나리오의 에러 핸들링 금지.
  200줄을 50줄로 줄일 수 있으면 다시 쓴다.
- **한 가지 방법**: 같은 값을 계산하는 경로는 시스템 전체에서 하나.
  적합도=`engine.score()`, %=`report._pct()`, 집계=`vote_counts_view`. 라우터에서 재계산 금지.
  (근거 실사고: 산식 이원화 → 순위 역전. 경로가 둘이면 반드시 어긋난다.)
- **대체 없음**: "혹시 몰라서" 남기는 구형 경로 금지.
  (알고리즘 폴백은 예외 — 환경 의존 대응이며 결과 동일성 검증됨.)
- **호환성보다 명확성**: 해커톤 코드에 하위 호환 부담 없음. 낡은 형식은 감싸지 말고 고친다.
- **빠른 실패**: 전제 깨지면 즉시 명확한 에러. 조용한 기본값이 틀린 배치를 만드는 게 최악
  (예: competitors JSON 파싱 실패 = 에러이지 빈 dict가 아님).
- **기본 메커니즘 신뢰**: SQLite 트랜잭션·FastAPI 검증을 재구현하지 않는다.
  자체 백업·자체 캐시·자체 ORM 금지.
- **관심사 분리**: 라우터는 얇게(입출력), 계산은 엔진, DB는 db.py. 함수는 단일 책임.
- 단, **정직성 장치(출처 태그·SEED·DISCLAIMER·집계 필터)는 단순화 대상이 아니다.**
  여기서의 중복·과함은 미덕이다.

## 6. 외과적 수정 (Surgical Changes)

꼭 고쳐야 할 것만 건드린다. 내가 만든 것만 치운다.
- 인접 코드·주석·포맷 "개선" 금지. 안 깨진 것 리팩토링 금지.
- 기존 스타일 준수: 한국어 주석, 확정/미확정 표기 관례, TEXT형 id("cafe", "V-A").
- 관련 없는 데드코드는 **언급만 하고 지우지 않는다.** 내 변경이 만든 미사용만 정리.
- 모든 변경 라인이 사용자 요청에 직접 추적되어야 한다.

## 7. 형사 원칙 — 증거 기반 디버깅 (Detective Rule)

**당신은 형사다. 버그는 범죄다.** 이론(가설)을 세우고, 증거를 수집하고,
증거가 이론을 증명한 뒤에만 수정한다. 증거 없는 아름다운 이론으로 무작위 수정 금지.
- 수정 전에 **재현**한다. 재현 못 하면 원인을 안다고 말하지 않는다.
- 로깅은 최소·표적(용의자 주변만). 수사 끝나면 내가 심은 로그 회수.
- **증상이 아니라 근본 원인**:
  - 실사례: POST /votes 500 → 증상 수정=try/except, 근본 원인=기존 DB에 voter_name 컬럼
    부재(마이그레이션 누락). 후자를 고친다.
  - 실사례: demo.py KeyError → 증상 수정=.get() 회피, 근본 원인=폐기된 가중합 형식 참조.
    최신 형식으로 고친다.
- 손계산이 최고의 증거: 산식 의심 시 test_engine의 손계산 케이스(V-A×카페=65.8)처럼
  종이 계산과 코드 출력을 대조.
- 증거가 두 이론 사이에서 갈리면 혼자 고르지 말고 증거와 함께 사용자에게 가져간다.

## 8. 목표 기반 실행 + 검증 (Goal-Driven + Verify)

다단계 작업은 계획 먼저:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
```
필수 검증(수정 후 매번):
- **컴파일**: `python -m py_compile` 바뀐 파일 전부.
- **회귀 4종**: test_engine(33) → test_step1(18) → test_step2(21) → test_step34(25).
  전부 통과 전에 "됐다"고 말하지 않는다. 실서버 기동 포함이므로 실제로 돌린다.
- **산식·표시 변경 시**: 단조성(점수 순위=% 순위) + 순위 역전 반례(10×1.0 vs 100×0.2) 재확인
  + 리포트 1순위=배정 업종 일치.
- **스키마 변경 시**: 기존 DB 파일 대상 마이그레이션 동작 확인(새 DB만 만들고 끝내지 않음).
- **정직성 스캔**: 금칙어 grep(투자|펀딩|수익 보장|성공 보장), SEED/is_seed 표기,
  리포트 값 출처 태그 존재.
- **환경 확인**: scipy 유무 양쪽에서 배치 결과 동일성.

## 14. 규제·책임 (코드 외적 주의)

- 1,000원 = 투자가 아니라 **쿠폰 선구매**. 불발 시 환불(모의 캐시 적립).
  이 프레이밍이 유사수신 회피 설계의 핵심 — 코드 문구가 이를 깨면 안 된다.
- 결제·환불·캐시 전부 모의. 실PG·실카드·사업자 필요 기능 금지.
- 개인정보 최소 수집(voter_name ≤30자 표시명만). 주소·연락처 추가는 개인정보 검토 선행
  (거주 분포 기능이 보류인 이유 중 하나).

- AI는 법률·투자 자문 자격이 없다. 리포트는 후보 필터 + 판단 재료. DISCLAIMER가 고지.
