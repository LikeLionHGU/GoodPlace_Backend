# 08. 리포트·공실 등록 확정 명세 (동네 투표 전환 반영)

**2026.07.14 확정.** 투표 대상이 '공실'에서 '동네+업종'으로 바뀌면서 수요·리포트·공실 등록이 모두 재정의됐다. 이 문서가 그 확정 기준이며, 03(산식)·07(리포트)·AB공통(스키마)의 관련 부분을 이 문서로 갱신한다.

## 0. 무엇이 바뀌었나 (핵심)

- **주민 투표 = 동네 + 업종.** 특정 공실이 아니라 "이 동네에 이 업종이 있으면 좋겠다"에 투표. 투표 시점의 GPS 위치를 **행동 범위**로 저장(주거지 주장 아님 — "이 동네를 실제로 지나다니는 소비자"의 신호).
- **공실↔업종 매칭 = 리포트 생성 시점에 계산.** 수요 수집과 배치가 시간적으로 분리.
- **리포트의 주어가 뒤집힘:** (기존) "이 공실에 어느 업종" → (신규) **"이 업종에 어느 공실"**.
- **창업자 진입:** 주민과 같은 동네 수요 리스트(업종별 득표 순위)를 봄 → 업종 클릭 → 그 업종의 추천 공실 + 리포트.

## 1. 투표자 위치 — 표현과 저장 (근거 게이트)

- **표현:** "행동 범위 기반 수요"(주거지 아님). 발표·리포트 문구도 이 표현으로 통일.
- **저장:** GPS는 정밀하게 받되(사용자 편의), **200m 격자로 반올림해 저장.** 정밀 좌표 원본을 쌓지 않는다.
  - 이유: 산식은 "반경 내 카운트"만 쓰므로 격자로 충분하고, 정밀 좌표 축적은 "누가 언제 어디" 민감정보가 된다.
  - 출처 태그: 투표 위치 기반 값은 '파생'.

## 2. 거리 감쇠 수요 (산식 재정의 — 03번 대체)

기존 `demand = 직접 투표수 + w1 × 인근 투표수`는 **폐기**(공실 직접 투표 개념이 사라짐).

```
공실 수요 demand = Σ (반경 내 각 투표자의 거리 가중)

거리 가중 w(d):
  d ≤ 500m           → 1.0            (핵심 생활권, 온전한 1명)
  500m < d ≤ 800m    → 1.0 − (d−500)/300   (선형 감소: 650m=0.5, 800m=0)
  d > 800m           → 0              (수요로 세지 않음)
```

- 대상 투표 = 그 공실이 속한 동네 + 해당 업종에 투표한 사람들.
- `score = demand × area_fit × competition_factor`의 나머지 두 항(면적·경쟁)은 유지.
- **w1은 폐기**(직접/인근 구분 소멸). w2=1.0(경쟁계수)은 유지.
- 500·800·선형은 **설계 초기값** — 양덕동 실투표 분포 확보 후 조정 조건. 격자 200m는 이 반경의 약수.
- breakdown 노출: 반경 내 투표자 수(원시), 거리 가중 합(demand), 500m 이내 인원 등.

## 3. 창업 리포트 내용 (07번 대체 — "이 업종에 어느 공실" 구조)

창업자가 동네 수요 리스트에서 **업종을 클릭**하면 뜨는 리포트. 업종은 이미 정해진 상태.

### 상단 — 업종 단위 (공실 무관)
1. **한 줄 결론** — "양덕동 세탁소, 추천 공실 3곳"
2. **동네 전체 수요** — "양덕동에서 세탁소를 원한 주민 32명" (동네 단위, 우리 고유 데이터. 특정 공실 아님)

### 본문 — 추천 공실 순위 (탭 또는 카드로 1·2·3위)
탭의 의미 = **그 업종에 추천하는 공실 순위**(기존 '업종 순위'에서 바뀜). 순위는 allocate가 아니라 그 업종에 대한 공실별 score 정렬(단일 업종이므로 겹침 문제 없음). 각 공실마다:

- **적합도 %** — 포화 곡선 변환(K 규칙 유지)
- **행동 범위 수요** — 이 공실 반경 내 이 업종 투표자 수 + 거리 가중 합. "동네 32명 중 이 자리 반경엔 18명" (공실마다 다름 = 위치 추천의 핵심 근거)
- **경쟁 상황** — 반경 500m 내 동일 업종 수 + 동네 평균 대비 비율(기회/주의/포화 플래그)
- **면적 적합** — 공실 면적이 업종 요구 범위에 맞는지
- **업종 가능 여부** — 층·용도·설비(§4 공실 등록 정보 기반). 미등록 항목은 "확인 필요"로 표시
- **대기 고객** — 이 공실 반경 내 투표자 수 = 개업일 첫 손님 후보 + 쿠폰 확정 가치(인원 × 3,000원)

### 하단 — 공통 (공실 무관)
- **참고 비용** — 업종 평균 초기비용(출처 + 단서, 정밀 예측 아님)
- **필요 인허가** — 업종별 신고·허가
- **"이 리포트가 답하지 않는 것"** 체크리스트 — 권리금·실임대조건·현장 관찰·상세 설비
- **DISCLAIMER** 고정 + 중개사 연결 CTA

### 빠지는 것 / 대체
- **유동인구:** 여전히 데이터 없음. 그러나 "행동 범위 기반 수요"가 그 자리를 대체 → 이 전환의 최대 수확(유동인구 추정치 대신 실제 지갑 연 주민 수를 셈).
- 정보 과잉(시간대별·성별연령·비용 분해) 배제 유지.
- 보류 3건(지역 특성·거주 분포·월 고정비) 유지. 단, 위치를 격자로 저장하게 되어 '거주 분포'는 데이터 축적 후 재개 검토 대상에 근접.

## 4. 공실 등록 항목 (중개사 입력 — 신규 확정)

공실 정보의 출처가 둘로 나뉜다. 소상공인 API로 얻는 것과 **중개사가 등록 시 입력**하는 것. 후자가 "업종 가능 여부" 판정의 근거가 된다(직방·다방이 매물 옵션을 등록받는 방식과 동일).

### API/실측으로 채우는 필드 (기존)
- 위치(lat/lng), 주소, region_code, 면적(area_m2), 직전 업종(prev_industry), 공실 기간, 경쟁점(competitors)

### ★ 중개사 등록 필드 (신규 — 출처 태그 '중개사등록')
- **층(floor)** — 1층 / 2층 이상 / 지하 (업종 적합·리포트 참고)
- **용도지역/건물 용도(building_use)** — 근린생활시설 여부 등. 중개사가 아는 범위로 입력, 보강은 §5
- **간이 설비 여부(facilities)** — 업종별 체크박스형:
  - 상하수도·급배수 (음식·세탁 계열 필수)
  - 환기·후드 가능 (음식 계열)
  - 가스 인입 (음식 계열)
  - 화장실 유무
  - 주차 가능
  - ※ 상세·정밀 설비는 등록 대상 아님(현장 확인) — 체크박스 수준만
- **권리금 유무·임대 조건** — 있으면 표시(값 정확성은 중개사 책임, 출처 태그 명시). 없으면 '문의'

### 업종 가능 판정 3층 구조 (근거 게이트)
1. **면적 적합** — API/실측 면적 vs 업종 요구 범위. ✅ 지금 가능
2. **층·용도·간이 설비** — 중개사 등록값. ✅ 등록되면 가능(출처 '중개사등록'), 미등록이면 "확인 필요"
3. **상세 설비·정밀 요건** — 🚫 데이터 없음. 리포트 "직접 확인" 항목으로 이관

## 4.5 AI 적용 확정 — 리포트 자연어 해설

- **모델: GPT API(OpenAI) — 팀 보유 키 사용 확정.** 경량 모델(mini 계열)로 시작, 품질 부족 시 상향. 비용은 리포트 1건 2~3천 토큰 수준이라 시연 규모에서 무시 가능.
- **역할:** §3 리포트의 확정 수치(수요·경쟁·면적·적합도 등)를 받아 창업자용 해설 3~5문장 생성. 공실 탭마다 1회.
- **환각 억제(근거 게이트의 LLM 판):** 입력 = 확정 수치 JSON만. 시스템 지시에 "이 수치 외 새 숫자·업종·사실 생성 금지, 성공 보장 표현 금지" 고정. 출력의 숫자를 추출해 입력에 없는 숫자면 폐기·재생성(숫자 화이트리스트 검사). 실패 시 해설 없이 수치 리포트만(조용한 대체 금지).
- **출력 태그:** 해설 문장은 'AI생성(수치는 원출처)' 태그.
- **키 관리:** `.env`에만. 코드·로그·저장소·채팅 평문 금지.
- AI 프레이밍(발표): 구조화(stretch) · 최적화(헝가리안·거리감쇠) · **생성(이것, 구현)** · 예측(로드맵).

## 5. 확정·미확정 정리

**확정:**
- 투표 = 동네+업종, 위치는 행동 범위·200m 격자 저장
- 거리 감쇠: ≤500m 가중 1.0, 500~800m 선형 감소, >800m 제외 (설계 초기값)
- 리포트 = 업종 클릭 → 추천 공실 순위, 탭 = 공실 순위
- 공실 등록에 중개사 입력 필드(층·용도·간이 설비·임대 조건) 추가, 출처 태그 '중개사등록'
- 업종 가능 판정 3층 구조(면적 / 중개사등록 / 확인필요)

**미확정 (확정처럼 쓰지 말 것):**
- 500·800·선형의 실데이터 검증 (양덕동 투표 분포 확보 후)
- 용도지역을 브이월드/건축물대장 API로 보강할지 (중개사 입력으로 시작, API 보강은 이후)
- 중개사 등록 UI를 MVP에 넣을지 / 시드로 대체하고 시연할지
- 캐시 정책 3건, K 전역/캠페인별 (기존 미확정 유지)
- OpenAI 키를 .env로 이관·권한 관리 (보유 키 확인됨 — 발급 불필요)

## 6. 스키마 영향 (AB공통 v3 반영 필요)

```
votes:  vacancy_id 제거 → region_code + industry_id + voter_grid(200m 격자 좌표)
        (B는 "동네+업종별, 격자별 투표자 수" 집계로 읽음)
vacancies: + building_use(용도), + facilities(JSON 체크박스), + rent_terms(임대조건),
           + source_tag(API/실측/중개사등록 구분)   ※ floor는 기존 존재
```
- **주의:** votes에서 vacancy_id 제거는 큰 계약 변경. A·B 합의 + 마이그레이션 필수.
- B의 vote_counts 집계 계약이 `(vacancy_id, industry_id)→수` 에서
  `(industry_id, grid)→수` 로 바뀜 → 공실 수요는 "공실 반경 내 grid들의 가중 합"으로 계산.

## 7. 성사 판정 — "우리를 통한 창업" 구분 (신규 확정)

BM(성사 수수료)·쿠폰 사용처 제한·전환율 예측 라벨의 전제. 세 겹 판정:

1. **성사 레코드(placements 신설):** `id, vacancy_id, industry_id, founder_id, campaign_id, confirmed_by(중개사/운영자), confirmed_at, status`. 중개사 연결 요청 → 계약 성사 확정 시 기록. MVP는 관리자 확인 버튼(모의)으로 시연.
2. **쿠폰 사용 = 사후 증거:** 쿠폰은 placements에 있는 매장에서만 사용 가능(시스템 강제). cash_ledger의 coupon 사용에 `ref_id → placement` 연결.
3. **API 대조 = 우리 밖 구분:** 소상공인 API 신규 개업(`opbizDt`)과 대조 → 우리 통한 창업(레코드 有) / 자연 창업(API 신규 점포, 레코드 無) / 미창업(둘 다 無). API 갱신 주기·주소 매칭 정확도는 첫 호출 검증 항목.

전환율 예측(로드맵)의 라벨 = placements + 해당 매장 쿠폰 회수 수. 이 구분 없이는 데이터 자산 서사가 성립하지 않음.
