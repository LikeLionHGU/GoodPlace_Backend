# 명당 백엔드 — 구현 현황 (단계별 코드·알고리즘·AI 적용)

> 기준: `code/` 실제 코드. 문서와 코드가 다르면 코드가 우선(CLAUDE.md 1절).
> 작성 시점 기준 완료 범위 = **1·2단계 + 3·4단계 통합**. 5단계(리포트 노출)·6·7단계는 미착수.
> 용어 규칙: 투자·펀딩 금지 → **선결제·쿠폰 선구매**. 결제는 전부 **모의**(실PG 없음).

**아키텍처 한 줄:** DB(SQLite) → **어댑터**(row를 엔진 입력으로 변환) → **순수 함수 엔진**(산식·배치) → API(FastAPI). 엔진은 DB를 모르고, DB는 엔진을 모른다. 둘을 잇는 얇은 층이 어댑터·라우터다.

```
[클라이언트] ──HTTP──> [FastAPI 라우터] ──> [어댑터] ──> [엔진(순수함수)]
                          routes_*.py       adapters.py    engine.py / report.py
                                              │
                                        [SQLite: mydang.db]  db.py
```

## 1단계 — DB·시드·조회 API (데이터가 '들어오는' 쪽)

### 만든 것
- **`db.py`** — SQLite 스키마 정의·연결·시드 적재. 테이블 5종: `industries`, `vacancies`, `votes`, `cash_ledger`, `campaign` + 집계 뷰 `vote_counts_view`.
  - 계약(AB공통) 스키마 그대로 + 추가 컬럼 3건: `industries.licenses`(리포트용 인허가), `campaign.is_seed`, `votes.voter_name`(고객 명단용). *← 계약 밖 추가라 A/문서 통지 대상.*
- **`routes_query.py`** — `GET /health`, `GET /industries`, `GET /vacancies`(+투표 요약·`region_code` 필터), `POST /admin/seed/reset`.
- **`main.py`** — FastAPI 앱. 기동 시 테이블 생성 + 최초 1회 자동 시드.
- **`config.py`** — 상수 한 곳 모음(2단계에서 캐시 정책도 여기 추가).

### 알고리즘·AI 적용
- **없음(의도적).** 이 단계는 순수 데이터 계층이다. AI/최적화가 들어가는 곳이 아니라, **엔진이 먹을 정직한 데이터를 준비**하는 단계.
- 핵심 설계 결정: 지역을 하드코딩하지 않고 **`region_code`로 파라미터화**(양덕동 → 경북 확장). 모든 시드는 `is_seed=1`/`SEED` 표기(가상 데이터 지양 요건).
- **B(엔진)와의 유일한 접점 = 집계 뷰.** `vote_counts_view`가 `(vacancy_id, industry_id) → 투표수` 형태를 보장한다. 엔진은 votes 원본 구조를 모르고 이 형태로만 읽는다(계약).

## 2단계 — 투표(모의 결제)·집계·캐시 뼈대

### 만든 것
- **`db.py`(추가)** — `create_vote`(1표=1,000원 서버 고정, `payment_status='held'`), `votes_summary`(전체·공실별·시드/실 구분), `cash_add`/`cash_balance`(캐시 원장 뼈대), `NotFound` 예외.
- **`routes_vote.py`** — `POST /votes`(없는 공실/업종 404, `voter_name` ≤30자, 어뷰징 방지는 MVP 밖으로 명시), `GET /votes/summary`.
- **`config.py`(추가)** — `CASH_POLICY` 3건 전부 **미확정 — 팀 결정 대기**로 표기(교환비율·유효기간·재투표 동일취급).

### 알고리즘·AI 적용
- **집계 로직**(AI 아님, 그러나 산식의 입력을 만드는 급소):
  - 수요로 세는 표 = `payment_status IN ('held','settled')`. **환불(`refunded`)·캐시전환 표는 집계에서 제외**(`COUNTED_STATUSES`). 이 제외 조건이 6단계 환불 규칙과 엔진 산식 정확도의 전제.
- **모의 결제:** 서버는 상태(`held`)만 기록. 실제 PG 호출 없음. 환불·정산 전이는 6단계.
- **핵심 논지 데이터:** 이 단계가 만드는 "이미 지갑을 연 유료 수요(투표)"가 곧 산식의 `demand` 입력이자, 5단계 리포트의 무게중심("이미 지갑 연 대기 고객 N명")이다.

### ⚠️ 알려진 이슈 (미해결)
- `votes.voter_name` 컬럼을 2단계에서 추가했으나, `db.py`가 `CREATE TABLE IF NOT EXISTS`라 **1단계에 이미 생성된 `mydang.db`에는 컬럼이 반영되지 않는다.** 기존 DB로 `POST /votes` 시 500(`no column named voter_name`), `POST /admin/seed/reset`으로도 복구 안 됨(DB 파일 삭제만이 복구). 새 DB에서는 정상. → **경량 마이그레이션(ALTER TABLE) 필요.**

## 3단계 — 수요 적합도 산식 (엔진, `engine.py` 확정본)

### 코드
`score()`, `area_fit()`, `competition_factor()`, `demand()`, `neighborhood_avg_competitors()`. 전부 **순수 함수**(인자로 받은 것만 계산, DB 접근 없음).

### 적용한 산식·알고리즘 (이것이 이 프로젝트의 "AI"의 1층)
```
score = demand × area_fit × competition_factor
```
- **demand(수요)** = `직접 투표수 + w1 × 인근 공실(반경 500m 이내)의 같은 업종 투표수`
  - '인근' 판정은 **하버사인(Haversine) 거리** 계산(`_haversine_m`)으로 위경도 → 미터 환산.
  - `w1 = 0.4` — 직접 표는 1차 신호, 인근 표는 절반 이하의 보조 신호.
- **area_fit(면적 적합)** = 면적 범위 안 1.0, 벗어나면 비율 감점(하한 0.2). 단조 감소.
- **competition_factor(경쟁 계수)** = **동네 평균 대비 비율** 기반.
  - `ratio = 그 공실의 경쟁점 수 / 그 업종의 동네 평균 경쟁점 수`
  - `ratio ≤ 1` → 1.0(감점 없음), `ratio > 1` → `1/(1 + w2×(ratio−1))`
  - `w2 = 1.0` — 동네 평균의 2배에서 수요 반토막(0.5). **절대 경쟁점 수로 감점하지 않는 이유:** "카페 3곳(흔함) ≠ 세탁소 3곳(드묾)". 업종별 정상 밀도를 존중하고 명백한 포화만 감점.

### 성격 (정직성 — 과장 금지)
- 이 산식은 **매출·성공을 예측하는 모델이 아니다.** "이 자리를 후보로 볼지" 거르는 **1차 후보 필터 신호**다. 진짜 창업 근거는 5단계 리포트 전체가 만든다.
- 심사 방어용으로, 응답에 항상 **점수 분해(breakdown)**를 동봉: `direct_votes`, `nearby_weighted`, `area_fit`, `competitor_count`, `neighborhood_avg`, `competition_ratio`, `competition_factor`.

## 4단계 — 겹침 해소 배치 (엔진, `engine.py` 확정본)

### 코드
`allocate()` + 배정 전략 3종: `_assign_hungarian`, `_assign_exhaustive`, `_assign_greedy`.

### 적용한 알고리즘 (이 프로젝트 "AI"의 2층 = 최적화)
- 문제 성격: **같은 업종을 두 공실에 배정하지 않으면서 전체 적합도 합을 최대화** = 전형적인 **배정(assignment) 문제**.
- 채택: **헝가리안 알고리즘**(scipy `linear_sum_assignment`). 최대화 문제라 점수를 음수로 뒤집어 최소 비용 매칭으로 푼다.
  - 근거: 공실이 수십~수백이어도 최적해를 빠르게. 전수탐색은 소수에서도 폭발.
- **환경 자동 폴백**(어느 환경에서도 실행): scipy 있으면 헝가리안, 없으면 **전수탐색(최적 보장)** → 규모 초과 시 그리디.
  - *현재 개발 환경엔 scipy 미설치 → `algorithm="exhaustive(최적)"`으로 동작 중. 전수탐색도 최적해라 결과는 헝가리안과 동일. 대규모(경북 확장) 배포 시 성능을 위해 scipy 설치 권장.*
- 응답에 **차순위(runners_up)**·**사용 가중치(weights)**·**실제 사용 알고리즘(algorithm)**을 항상 실어 심사 방어.

### 리포트와의 일관성 (확정)
- 리포트 1순위 추천 = `allocate`가 배정한 업종. 지도 핀과 리포트가 같은 업종을 말하게(리포트가 공실별로 다시 점수 매기면 겹침이 되살아나므로 금지).

## 3·4단계 통합 — 어댑터 + 배치 API

### 만든 것
- **`adapters.py`** — DB row → 엔진 입력 dict 변환(얇은 연결층).
  - `vacancy_from_row`, `industry_from_row`, `load_allocation_inputs`.
  - **★ 최대 급소:** `competitors`는 DB에 **JSON 문자열**로 저장된다. `_parse_competitors`가 반드시 **dict로 파싱**해서 엔진에 넣는다. 문자열째 넘기면 경쟁점 수가 0으로 읽혀 **에러 없이 조용히 틀린 배치**가 나온다 → 파싱을 이 한 곳으로 강제.
- **`routes_allocation.py`** — `GET /allocation?region_code=...`(필수 인자). 어댑터로 모아 `engine.allocate()`를 **그대로** 호출, 산출을 축약 없이 반환. 응답 시간(`elapsed_ms`) 측정·로그.
- **`main.py`(추가)** — allocation 라우터 연결.

### 알고리즘·AI 적용
- 이 단계는 **새 알고리즘을 만들지 않는다.** 3·4단계에서 확정된 산식·배치를 **실데이터 흐름에 연결만** 한다(엔진 무수정).
- 유지한 계약: 투표 집계는 `(vacancy_id, industry_id)→투표수` 형태 그대로, **환불 제외** 상태로 엔진 투입.
- 검증(`test_step34.py`)에서 **DB 원본 손계산과 API breakdown이 일치**함을 숫자로 대조(예: fruit 동네평균 1.75, competition_ratio 1.14 등) → "조용히 틀림" 부재를 실증.

- **6단계**: 캠페인 상태·환불(캐시 적립 모의) 로직 — 미착수.

## "AI를 어떻게 적용하는가" — 정직한 정리

현재 엔진의 "AI"는 **머신러닝·LLM이 아니라, 실제 주민 유료 수요 데이터 위에서 도는 (1) 규칙 기반 수요 적합도 산식 + (2) 조합 최적화(헝가리안 배정)** 이다. 즉 **데이터 기반 의사결정 지원(decision support)** 이지, 매출·성공을 예측하는 예측 모델이 아니다.

- **왜 이게 의미 있나:** 남들이 '유동인구·상권 통계'라는 간접 지표로 추정할 때, 명당은 **"이미 1,000원을 선결제한 유료 수요"** 라는 직접 신호를 산식의 입력으로 쓴다. AI(최적화)는 그 수요를 빈 상가에 **겹침 없이 최적 배치**하는 역할.
- **과장하지 않는 선:** 산식은 후보 필터일 뿐이고, 창업 근거의 무게중심은 리포트 전체(특히 대기 고객 수)에 있다. 예측·보장 표현은 쓰지 않는다.
- **향후 AI 확장 여지**(현재 미구현·미확정): 실데이터(7단계) 확보 후 수요 예측·업종 추천에 학습 기반 모델을 얹을 수 있으나, 지금 코드는 **결정적(deterministic) 알고리즘**으로만 되어 있다.
