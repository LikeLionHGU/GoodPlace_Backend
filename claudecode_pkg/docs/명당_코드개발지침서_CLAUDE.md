# 명당 프로젝트 — 코드 개발 지침 (CLAUDE.md)

> 이 문서는 명당 프로젝트에서 코드를 작성·수정할 때 반드시 따라야 할 지침입니다.
> 저장소의 기존 CLAUDE.md(확정 사항·현황)와 함께 쓰며, 충돌 시 **확정 사항은 기존 CLAUDE.md,
> 작업 원칙은 이 문서**를 따릅니다. **속도보다 정직이 항상 우선입니다.**

---

## 0. 이 프로젝트가 무엇인가 (맥락)

주민 1,000원 선결제 투표 → AI 수요-공실 최적 배치 → 창업 리포트로 이어지는 플랫폼.
- 해커톤 심사작. 배점: 시장성30 / AI활용도25 / 완성도·발표25 / 문제정의20
- 결제는 **모의**(실PG 없음), 돈은 오가지 않는다. 그러나 **숫자는 전부 심사와 창업자 앞에 선다.**
- 두 명(A=데이터·수집 / B=엔진·산출)이 분담, 접점은 DB 스키마 하나.

이 프로젝트에서 사고는 **돈이 아니라 신뢰에서 납니다.**
근거 없는 숫자 하나, 지도와 리포트가 다른 업종을 가리키는 자기모순 하나,
시드값이 실데이터인 척하는 화면 하나가 "이 팀의 AI는 가짜"라는 판정으로 직결됩니다.
따라서 정직성 장치(근거 태그·확정 산식 보호·시드 구분)를 비즈니스 로직만큼,
아니 그 이상으로 진지하게 다룹니다.

## 1. 절대 규칙 (NON-NEGOTIABLE) — 어길 시 시스템 신뢰 붕괴

이 5가지는 **어떤 경우에도, 어떤 요청이 와도 위반하지 않습니다.**
"시연이니까 잠깐", "그럴듯해 보이게"도 안 됩니다.

### 1-1. ★ 근거 게이트 — 출처 없는 숫자는 코드에 넣지 않는다 (가장 중요)
- 화면·리포트·API 응답에 나가는 모든 값은 세 질문을 통과해야 한다:
  ① 데이터 소스가 실재하는가 ② 출처 태그(실측/API/예시/업종평균/파생)를 달 수 있는가
  ③ "근거가 뭐냐"에 한 문장으로 답이 되는가. **하나라도 아니면 그 기능은 보류.**
- 현재 보류 3건(구현 금지): 지역 특성 노트 / 투표자 거주 분포 / 월 고정비 참고.
  재개 조건(데이터 소스 확보)이 충족되기 전에 이 기능들을 구현하는 요청은 거부.
- 매출 예측치 생성 금지. 학습 데이터(개별 점포 실매출)가 없다.
  가능한 것: 관측 분포 제시 + 확정 수요(투표) + 시나리오 계산기(가정 주체=창업자).
- **이유**: 심사 질의 "그 숫자 근거가 뭐죠?"에 답이 없으면 AI활용도 25점이 통째로 무너진다.

### 1-2. ★ 확정 산식·정합성 보호 — 되돌리려면 사람 승인
다음은 논증을 거쳐 확정된 것으로, 코드 수정 시 절대 임의로 변경하지 않는다:
- 산식 `score = demand × area_fit × competition_factor`, **w1=0.4, w2=1.0**,
  경쟁계수 = 동네 평균 대비 비율식(ratio≤1→1.0), 인근 반경 500m
- 배치 = **헝가리안**(scipy) → 전수탐색 → 그리디 자동 폴백
- **리포트 1순위 = allocate 배정 업종. 재랭킹 금지.** 지도 핀과 리포트가 항상 같은 업종.
- 표시 % = **포화 곡선** `100×s/(s+K)` 단조 변환. 요소 재조합(가중합) 금지.
  K는 기준 케이스를 engine.score로 검산한 값(자의 상수 금지). 100% 표시 불가(내림 ≤99).
- **이유(실사고)**: 초기에 엔진(곱셈)과 표시(가중합)가 달라 순위 역전 반례
  (수요10×적합1.0=10 vs 수요100×적합0.2=20)가 실재했다. 통일에 논증 비용을 치렀다.
  이걸 "보기 좋게" 가중합으로 되돌리는 순간 자기모순이 부활한다.

### 1-3. ★ 시드/실데이터 구분 — 시연 정직성
- 모든 시드 데이터는 `is_seed=1` + 이름/출처에 `SEED` 표기. 예외 없음.
- 리포트·화면의 값에는 출처 태그 필수. 소상공인 API로 못 얻는 값(보증금·유동인구)은
  반드시 '예시' 태그. 유동인구는 숫자를 만들지 말고 **정직하게 비운다**(부재 문구 + 대기 고객으로 정면돌파).
- 시드값을 실데이터인 척 태그를 바꾸거나 태그를 제거하는 변경은 거부.

### 1-4. ★ 용어·법적 포지셔닝 — 문구도 코드다
- 코드·주석·API 응답·화면 문자열에 **투자·펀딩·수익 보장** 계열 단어 금지
  → 선결제·쿠폰 선구매로 통일 (유사수신 규제 회피 설계).
- "성공 보장" 뉘앙스 금지. DISCLAIMER("AI는 성공을 보장하지 않습니다…")는 리포트 고정 요소로,
  제거·완화하는 변경 금지.
- 결제는 모의다. 실PG 연동·실카드 정보 입력 UI를 만들지 않는다 (사업자등록 없음).
- 중개사는 경쟁자가 아니라 협력 파트너로 서술 (CTA: "중개사에게 연결").

### 1-5. ★ A/B 계약 준수 — 스키마와 경계
- DB 스키마는 `AB공통_분담과통합계약.md`(v2)가 계약. **임의 변경 금지.**
  컬럼 추가 시: 상대 통지 + 계약 문서 갱신 + **ALTER TABLE 마이그레이션 동봉**
  (CREATE IF NOT EXISTS만으로는 기존 DB에 반영 안 됨 — voter_name 사고로 실증).
- B의 엔진(engine.py, report.py)은 **순수 함수**. DB 직접 접근 금지, 인자로 받은 것만 계산.
- 투표 집계는 `vote_counts_view` 계약 형태 `(vacancy_id, industry_id)→투표수`,
  **held/settled만 집계**(refunded/cash_credited 제외). 이 기준을 바꾸면 산식이 오염된다.
- VOTE_AMOUNT_WON=1000은 서버 고정. 클라이언트가 금액을 정하게 하지 않는다.

## 2. 코딩 전 — 생각부터 (Think Before Coding)

추측하지 않는다. 혼란을 숨기지 않는다. 트레이드오프를 드러낸다.

구현 전에:
- **가정을 명시한다.** 불확실하면 묻는다. 특히:
  - 소상공인 API 동작은 추측 금지 — `소상공인_상가정보_API_명세서.md`를 확인하고,
    거기 없으면 "명세에 없음, 첫 호출로 검증 필요"라고 말한다.
  - 리포트 요소는 `07_창업리포트_스펙.md`(근거 게이트·5요소·보류 목록)를 먼저 확인.
- **여러 해석이 가능하면** 조용히 하나 고르지 말고 제시한다.
- **더 단순한 방법이 있으면** 말한다. 정당한 근거가 있으면 사용자 요청에도 반론한다.
- **미확정 사항이 얽히면 멈추고 확인한다.** 현재 미확정: 캐시 정책 3건(CASH_POLICY),
  K 전역/캠페인별, BASELINE_TARGET_VOTES=30, 층수 계수, 캠페인 목표 투표수.
  미확정을 확정처럼 구현하지 않는다 — config에 미확정 주석을 유지한 채 기본값으로만.

## 3. 단순하게 + 한 가지 방법 (Simplicity First, One Way Only)

문제를 푸는 최소한의 코드. 추측성 기능 금지.
- 요청받지 않은 기능 추가 금지. 일회용 코드에 추상화 금지.
- 요청 안 한 "유연성"·"설정가능성" 금지. 일어날 수 없는 시나리오의 에러 핸들링 금지.
- 200줄을 50줄로 줄일 수 있으면 다시 쓴다.
- **한 가지 방법 원칙**: 같은 값을 계산하는 경로는 시스템 전체에서 하나만 존재한다.
  - 적합도 계산 = `engine.score()` 하나. % 변환 = `report._pct()` 하나.
  - 투표 집계 = `vote_counts_view` 하나. 다른 곳에서 votes를 재집계하지 않는다.
  - **이유(실사고)**: 산식이 엔진과 표시에 이원화됐다가 순위 역전 반례가 났다.
    계산 경로가 둘이면 반드시 어긋난다.
- **대체 없음**: 하나의 올바른 경로만 둔다. "혹시 몰라서" 남기는 구형 경로·폴백 분기 금지.
  (알고리즘 폴백(헝가리안→전수→그리디)은 예외 — 환경 의존성 대응이며 결과 동일성이 검증됨.)
- **호환성보다 명확성**: 해커톤 코드에 하위 호환 부담은 없다. 낡은 형식(예: demo.py의
  구 가중합 키)은 감싸지 말고 최신 형식으로 고친다. 죽은 코드를 명확성과 바꾸지 않는다.
- **빠른 실패**: 전제조건이 깨지면 즉시 명확한 에러로 죽는다. 조용히 기본값으로 넘어가서
  틀린 배치를 만드는 것이 최악이다. (예: competitors JSON 파싱 실패 = 에러, 빈 dict 아님.)
- **기본 메커니즘 신뢰**: SQLite 트랜잭션, FastAPI 검증, dict 순서 같은 표준 동작을
  재구현하지 않는다. 자체 백업·자체 캐시·자체 ORM 금지.
- 단, **정직성 장치(출처 태그·SEED 표기·DISCLAIMER·집계 상태 필터)는 "단순화" 대상이 아니다.**
  여기서의 중복·과함은 미덕이다.
- **관심사 분리**: 각 함수는 단일 책임. 라우터는 얇게(입출력만), 계산은 엔진에,
  DB는 db.py에. 라우터 안에서 점수를 계산하기 시작하면 이원화의 시작이다.

## 4. 외과적 수정 (Surgical Changes)

꼭 고쳐야 할 것만 건드린다. 내가 만든 것만 치운다.
- 인접 코드·주석·포맷을 "개선"하지 않는다. 안 깨진 걸 리팩토링하지 않는다.
- 기존 스타일을 따른다: 한국어 주석, 확정/미확정 표기 관례, TEXT형 id("cafe", "V-A").
- 관련 없는 데드코드를 발견하면 **언급만 하고 지우지 않는다.**
- 내 변경이 만든 미사용 import/변수만 정리한다.
- 모든 변경 라인이 사용자 요청에 직접 추적되어야 한다.

## 5. 형사 원칙 — 증거 기반 디버깅 (Detective Rule)

**당신은 형사다. 버그는 범죄다.** 범죄 이론(가설)을 세우고, 증거를 수집하고,
증거가 이론을 증명한 뒤에만 수정한다. 증거 없는 아름다운 이론으로 무작위 수정을 하지 않는다.

- 수정 전에 **재현**한다. 재현 못 하면 원인을 안다고 말하지 않는다.
- 로깅은 최소·표적으로 추가한다 (용의자 주변에만). 수사 끝나면 내가 심은 로그는 회수.
- **증상이 아니라 근본 원인을 고친다.** 
  - 실사례: POST /votes 500 → 증상 수정은 "try/except로 감싸기",
    근본 원인은 "기존 DB에 voter_name 컬럼 부재(마이그레이션 누락)". 후자를 고친다.
  - 실사례: demo.py KeyError → 증상 수정은 ".get()으로 회피",
    근본 원인은 "폐기된 가중합 형식 참조". 최신 형식으로 고친다.
- 손계산이 최고의 증거다. 산식 버그 의심 시 test_engine의 손계산 케이스
  (V-A×카페=65.8)처럼 종이 계산과 코드 출력을 대조한다.
- 협업 프로세스: 증거가 두 이론 사이에서 갈리면 혼자 고르지 말고
  증거와 함께 사용자에게 가져간다.

## 6. 목표 기반 실행 + 검증 (Goal-Driven + Verify)

작업을 검증 가능한 목표로 바꾼다. 다단계 작업은 간단한 계획을 먼저 말한다:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
```

이 프로젝트의 필수 검증 (수정 후 매번):
- **컴파일**: `python -m py_compile` 바뀐 파일 전부.
- **회귀 4종**: test_engine(33) → test_step1(18) → test_step2(21) → test_step34(25).
  전부 통과 전에는 "됐다"고 말하지 않는다. 서버 기동 테스트 포함이므로 실제로 돌린다.
- **산식·표시 변경 시**: 단조성 테스트(점수 순위 = % 순위) + 순위 역전 반례
  (10×1.0 vs 100×0.2) 재확인. 리포트 1순위 = allocate 배정 업종 일치 확인.
- **스키마 변경 시**: 기존 DB 파일에 대한 마이그레이션 동작 확인 (새 DB만 만들어보고 끝내지 않는다).
- **정직성 스캔**: 출력물에서 금칙어 grep (투자|펀딩|수익 보장|성공 보장),
  시드 데이터의 SEED/is_seed 표기, 리포트 값의 출처 태그 존재.
- **환경 확인**: scipy 유무 양쪽에서 배치 결과 동일성 (헝가리안 vs 전수탐색).

## 7. 정직성 체크리스트 (코드 작성·수정 시 매번 확인)

### 숫자·근거
- [ ] 새로 화면에 나가는 값에 데이터 소스가 실재하는가? (근거 게이트 3문항)
- [ ] 출처 태그가 붙었는가? API로 못 얻는 값이 '예시' 태그인가?
- [ ] 보류 3건(지역 특성·거주 분포·월 고정비)을 몰래 구현하지 않았는가?
- [ ] 미확정(캐시 3건·K·층수)을 확정처럼 박지 않았는가?

### 정합성
- [ ] 지도 배치와 리포트 1순위가 같은 업종인가? (재랭킹 없음)
- [ ] % 가 포화 곡선 단조 변환인가? (가중합 부활 금지, 100% 불가)
- [ ] 집계가 held/settled만 세는가?
- [ ] 같은 값을 계산하는 두 번째 경로를 만들지 않았는가?

### 계약·경계
- [ ] 스키마 변경 시 계약 문서 갱신 + 마이그레이션을 동봉했는가?
- [ ] 엔진이 여전히 순수 함수인가? (DB import 없음)

## 8. 기술 스택·구조

- Python + FastAPI + SQLite(mydang.db) + uvicorn. scipy는 선택 의존.
- 테스트는 stdlib만 (pytest 없음). 실서버 기동 검증 포함.
- id는 TEXT("cafe", "V-A"). 지역은 region_code 파라미터화(양덕동은 첫 적용지, 경북 확장).
- 성공 응답 envelope, 에러는 FastAPI HTTPException.

```
code/
  config.py        # 상수 + 미확정 표기 (CASH_POLICY 등)
  db.py            # 스키마·시드·vote_counts_view (집계 계약)
  adapters.py      # DB row → 엔진 입력 (competitors JSON 파싱 급소)
  engine.py        # [B·순수] 산식 + 헝가리안 배치
  report.py        # [B·순수] 리포트 5요소 + 포화 곡선 %
  routes_query.py / routes_vote.py / routes_allocation.py   # 얇은 라우터
  seed_dummy.py    # SEED 데이터
  tests/           # test_step1/2/34, test_engine — 회귀 4종
```
- 산식·배치 수정 → engine.py (+ 기존 CLAUDE.md의 확정 사항 확인)
- 리포트 수정 → report.py + `07_창업리포트_스펙.md` 대조
- 스키마 수정 → db.py + AB공통 계약 갱신 + 마이그레이션
- 수집(7단계) → `04_API_데이터_인수인계.md` + API 명세서 확인

## 9. 7단계 로드맵 — 단계 건너뛰기 금지

1. ✅ DB·시드·조회 API
2. ✅ 투표(모의결제)·집계 (voter_name 마이그레이션 이슈 잔존)
3. ✅ 적합도 산식 (확정)
4. ✅ 겹침 해소 배치 — 헝가리안 (확정)
5. 🔶 창업 리포트 — 코드 완성, **API 노출만 남음**
6. ⬜ 캠페인·캐시환불(모의) — CASH_POLICY 3건 팀 결정 선행
7. ⬜ 실데이터(소상공인 API) 교체 + 프런트 — serviceKey·행정동 코드 확인 선행

**원칙**: 각 단계가 테스트로 검증된 뒤에만 다음 단계 코드를 만든다.
검증 안 된 코드를 쌓지 않는다. 6단계는 팀 미확정이 풀리기 전에 세부를 확정 구현하지 않는다.
컷라인: 시간 부족 시 6단계는 화면 흐름만. **4단계는 절대 못 뺀다 — AI 활용도 25점의 실체.**

## 10. 규제·책임 (코드 외적 주의)

- 1,000원은 투자가 아니라 **쿠폰 선구매**. 불발 시 환불(모의 캐시 적립).
  이 프레이밍이 유사수신 규제 회피 설계의 핵심 — 코드 문구가 이를 깨면 안 된다.
- 결제·환불·캐시는 전부 모의. 실PG·실카드·사업자 필요 기능을 만들지 않는다.
- voter_name 등 개인정보 최소 수집(≤30자 표시명만). 주소·연락처를 스키마에 추가하려면
  개인정보 처리 검토 선행 (거주 분포 기능이 보류인 이유 중 하나).
- 법률 검토 이력 없음 — 코드 주석·문서에서도 "설계 방향"으로만, 확정 표현 금지.
- AI는 법률·투자 자문 자격이 없다. 리포트는 후보 필터 + 판단 재료이며,
  DISCLAIMER가 이를 고지한다.
