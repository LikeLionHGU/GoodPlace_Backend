# 명당 — 기획·개발 스토리 v2 (전면 개정판)

**2026.07.14 확정.** 동네 투표 전환·거리 감쇠·리포트 재편·중개사 등록·AI 자연어 생성을 반영한
**기획 서사의 단일 원본.** 기존 문서(06 스냅샷·01 개요·07 스펙)의 v1 서술과 충돌하면 이 문서가 우선.
기술 상세는 08(명세)·CLAUDE 통합본(개발 지침)이 담당.

---

## 0. v1 → v2 무엇이 바뀌었나 (한눈에)

| | v1 (구모델) | v2 (확정) |
|---|---|---|
| 주민 투표 대상 | 특정 빈 상가(공실) | **동네 + 업종** |
| 투표자 위치 | 없음 | **GPS 행동 범위** (200m 격자 저장) |
| 공실↔업종 매칭 | 투표 시점에 고정 | **리포트 생성 시점에 AI가 계산** |
| 리포트의 질문 | "이 공실에 어느 업종?" | **"이 업종에 어느 공실?"** |
| 창업자 진입 | 공실 선택 → 업종 추천 | **수요 리스트 → 업종 클릭 → 추천 공실** |
| 수요 산식 | 직접 투표 + w1×인근 | **거리 감쇠 가중 합** (w1 폐기) |
| 공실 정보 출처 | API/실측 | + **중개사 등록** (층·용도·간이설비·임대조건) |
| AI | 최적화(헝가리안)뿐 | + **LLM 리포트 자연어 생성** (확정) |
| 유동인구 | 약점 (정직한 부재) | **행동 범위 수요로 대체 — 강점 전환** |

바뀌지 않은 것: 문제 정의(5Whys), 1,000원 선결제 구조, 근거 게이트, 용어 규칙,
검증 현황(설문·인터뷰), 심사 배점 대응, 모의 결제·캐시 환불, 경북 확장 설계.

## 1. 문제 (유지)

경북 동네 상가 공실이 장기화되는 근본 원인은 임대료가 아니라 **수요 불확실성 + 집객**.
예비창업자가 "이 동네가 뭘 원하고 실제로 돈을 쓸지" 알 방법이 없다(정보 비대칭).
기존 상권분석은 과거 데이터, 주민 의견 채널은 무료 공감이라 소비 의사를 증명 못 한다.

## 2. 솔루션 한 줄 (v2)

**주민이 1,000원으로 '생기면 쓸 3,000원 쿠폰'을 선구매하며 "우리 동네에 이 업종"에 투표하면,
AI가 주민들의 행동 범위와 공실·경쟁 데이터를 겹쳐 그 업종이 성립할 최적의 빈 상가를 찾아
예비창업자에게 근거와 첫 손님 명단을 연결한다.**

## 3. 서비스 플로우 v2 (세 사용자)

```
[주민]   동네 수요 리스트를 봄 → 원하는 업종에 1,000원 선결제 투표
         (투표 순간의 GPS = 행동 범위. 200m 격자로 저장 — "이 동네를 실제로 다니는 소비자" 신호)

[중개사] 공실 등록: 위치·면적(API/실측) + 층·용도·간이설비·임대조건(중개사 입력)
         → 협력 파트너. 계약 성사 시 중개 수수료는 중개사 몫

[창업자] 같은 동네 수요 리스트(업종별 득표 순위)를 봄 → 업종 클릭
         → AI 추천 공실 1·2·3위 + 공실별 리포트(왜 이 자리인가)
         → 마음에 들면 중개사 연결 → 개업 시 선결제 고객이 첫 손님

[성사]   쿠폰 발급·첫 손님 확보   [불발] 캐시 적립(모의) → 재투표·쿠폰 구매
```

핵심 전환의 의미: 주민 행동이 자연스러워졌다(빈 상가 목록을 아는 주민은 없지만
"우리 동네에 뭐가 있으면"은 누구나 말할 수 있다). 그리고 공실 매칭이 투표가 아니라
**AI의 일**이 되어, AI의 역할이 서사적으로 선명해졌다.

## 4. AI가 들어가는 곳 (정직한 3단 프레임)

**"AI가 흩어진 주민 수요를 구조화하고, 위치 데이터로 최적 배치를 찾고, 결과를 창업자의 언어로 설명한다."**

1. **최적화 (현재 구현)** — 거리 감쇠 수요 × 면적 × 경쟁의 공실별 점수 + 헝가리안 배치.
   규칙 기반임을 숨기지 않고 '최적화'로 정직하게 부른다.
2. **생성 (이번 확정 — LLM)** — 리포트 자연어 해설. 우리가 계산한 **확정 수치만 입력**받아
   3~5문장 맞춤 해설을 생성. 새 숫자·사실 생성 금지(환각 억제 프롬프트).
   - 모델: **GPT API(OpenAI, 팀 보유 키 사용 확정)** — 경량(mini 계열)로 시작, 필요 시 상향.
     비용은 시연 규모에서 무시 수준(리포트 1건 2~3천 토큰).
   - 키 관리: 환경변수로만, 코드·로그·저장소·문서에 노출 금지.
3. **구조화 (로드맵)** — 자유 텍스트 수요("밤에 야식 먹을 데")의 업종 분류·군집화.
   여유 시 착수, 아니면 로드맵으로 발표.
4. **예측 (로드맵)** — 캠페인 누적 후 투표→실방문 전환율 학습.
   "카드사도 못 만드는 우리만의 학습 데이터" — 데이터 자산 BM과 연결.

## 5. 데이터와 근거 게이트 (v2 소스 맵)

| 데이터 | 출처 | 태그 |
|---|---|---|
| 업종별 동네 수요·투표자 행동 범위 | 자체 투표 (200m 격자) | 파생 |
| 공실 위치·면적·직전 업종·공실 기간 | 소상공인 API + 실측 | API/실측 |
| 경쟁점 수·동네 평균 | 소상공인 API 집계 | API |
| 층·용도·간이설비·임대조건 | **중개사 등록** | 중개사등록 |
| 업종 평균 창업비용·인허가 | 공단 통계·법령 | 업종평균/확인 |
| 리포트 해설 문장 | LLM (입력=위 확정 수치만) | AI생성(수치는 원출처) |

보류 유지(근거 게이트 미통과): 지역 특성 노트 / 월 고정비 참고 / 유동인구(→행동 범위 수요로 대체됨) /
상세 설비(→중개사 등록 밖 정밀 요건은 창업자 확인 항목).
'거주 분포'는 격자 데이터가 쌓이면 재개 검토(지금은 보류 유지).

## 6. 수요 산식 v2 (상세는 08번)

```
공실 수요 demand = Σ 투표자별 거리 가중   (같은 동네·같은 업종 투표자)
  d ≤ 500m → 1.0  /  500~800m → 1−(d−500)/300  /  > 800m → 0
score = demand × area_fit × competition_factor   (면적·경쟁 항 유지, w2=1.0 유지, w1 폐기)
표시 % = 포화 곡선 100×s/(s+K)  (단조 변환·요소 재조합 금지·100% 미도달 — 유지)
```
500/800/선형/격자 200m은 **설계 초기값** — 양덕동 실투표 분포 확보 후 조정 조건.

## 7. 창업 리포트 v2 (상세는 08번 §3)

업종 클릭 → ①한 줄 결론+동네 전체 수요(예: "양덕동 세탁소 32명") → ②추천 공실 1·2·3위
탭/카드(공실별: 적합도%·행동 범위 수요 "32명 중 이 반경 18명"·경쟁+포화 플래그·면적·
업종 가능 여부(중개사등록 기반, 미등록은 '확인 필요')·대기 고객+쿠폰 가치) →
③**LLM 해설 3~5문장**(확정 수치만) → ④참고 비용+인허가 → ⑤"답하지 않는 것" 체크리스트
+ DISCLAIMER + 중개사 연결 CTA.

## 8. 비즈니스 모델 (유지 + 강화)

수수료(성사 시 선결제금 일정 비율) / 점포 단위 심화 리포트 건당 / B2G 수요 리포트(POPEN 회신 대기).
v2 강화점: **업종×위치×행동범위 수요 지도**라는 데이터 자산이 더 정밀해짐 + 중개사 등록으로
공급측 데이터도 플랫폼에 쌓임 → 양면 데이터 플랫폼 서사.

## 9. 검증 현황 (유지) + 추가 계획

가설1(임대료 하방경직) 일부 참 / 가설2(1,000원 지불 의사) 참 — 26명 92.3%, 환불 구조가 결정 요인 /
가설3(수요 데이터 유용) 부분 참+보완. 반대 데이터(모모·스팸 우려) 병기 유지.
추가: 리포트 실전 검증 — 솔밭샤브·카피엔드에 v2 리포트 제시, "창업 결정에 충분한가" 확인.

## 10. 개발 현황과 전환 계획 (정직하게)

**현재 코드(mini-main)는 v1 모델 구현 상태다.** 1~4단계 완료·테스트 97개 통과·리포트 코드
완성(API 미노출)까지 전부 **공실 투표 기준**. v2 전환은 아래 작업을 거쳐야 하며,
**전환 완료 전까지 코드와 이 문서가 다른 것은 버그가 아니라 예정된 간극**이다.

전환 작업 목록 (순서대로, 단계별 검증 유지):
1. **스키마 마이그레이션** — votes: vacancy_id 제거 → region_code+industry_id+voter_grid.
   vacancies: +building_use, facilities(JSON), rent_terms, source_tag. (AB계약 v3 갱신 동반)
2. **집계 계약 변경** — vote_counts_view: (vacancy_id,industry_id)→수 에서 (industry_id,grid)→수 로.
3. **엔진 demand 재정의** — 거리 감쇠 합(08 §2). w1 제거. 기존 테스트의 손계산 케이스 재작성.
4. **리포트 재편** — build_report를 "업종→공실 순위" 구조로. 단일 업종이므로 배치 겹침 로직은
   리포트 경로에서 빠지고, 헝가리안은 '동네 전체 최적 배치 지도'용으로 존치.
5. **LLM 해설 연동** — GPT API(보유 키), 환각 억제 프롬프트(확정 수치 JSON만 입력), 키는 환경변수.
6. **동네 수요 리스트 API** — 주민·창업자 공용 (업종별 득표 순위).
7. 기존 이슈 3건(voter_name·demo·README)은 마이그레이션(1번)에 흡수 또는 선처리.

## 12. 발표 스토리 라인 (v2)

문제(공실=수요 불확실성) → 해법(지갑으로 증명하는 동네 수요 + 행동 범위) →
AI 3단(구조화·최적화·생성) → 시연(투표 → 수요 리스트 → 업종 클릭 → 추천 공실+LLM 리포트) →
정직성(근거 태그·보류 목록·DISCLAIMER — "우리는 모르는 것을 모른다고 표시한다") →
데이터 자산 로드맵(전환율 예측은 우리만 만들 수 있다).
유동인구 질문이 오면: "추정치 대신 실제로 이 동네를 다니며 지갑을 연 주민 수를 셉니다."
