# 담당 A 가이드 — 데이터·수집 파트

**먼저 `AB공통_분담과통합계약.md`를 읽고 이 문서를 본다.** 경계·스키마·통합 규칙은 공통 문서가 기준이다.

## A의 책임 한 줄

데이터가 **들어오는** 쪽 전부. DB를 만들고, 소상공인 API로 공실·점포를 수집하고, 주민 투표·모의결제·캐시를 처리한다. B는 A가 채운 데이터를 읽어 계산만 하므로, **데이터의 정확성과 스키마 준수가 A의 핵심 책임**이다.

## A가 만드는 단계

### 1단계 — DB·시드·조회 API
- 공통 문서의 스키마대로 `industries`·`vacancies`·`votes`·`cash_ledger`·`campaign` 생성
- 시드: 업종 6종(카페·베이커리·분식·서점문구·반찬·과일 — 면적·창업비용), 공실 몇 곳(양덕동, `is_seed=1`)
- 조회 API: 업종 목록, 공실 목록(공실별 투표 요약 포함)
- **B에게 줄 집계 함수/뷰 제공:** `(vacancy_id, industry_id) → 투표수`
- 검증: 시드 개수·면적범위(min<max)·좌표·경쟁점 JSON 정합성·SEED 표기·votes 빈 상태

### 2단계 — 투표·모의결제·캐시
- 투표 생성 API: 1,000원 고정, `payment_status='held'`(모의). 없는 공실/업종 404
- 투표 집계 API + 시드/실투표 구분
- **모의결제 흐름:** 실제 청구 없이 상태만(장바구니→결제수단→완료 화면 흐름은 프런트, 서버는 상태 기록)
- **캐시(모의):** 환불 시 현금 대신 `cash_ledger`에 +적립, 재투표·쿠폰구매 시 -사용. 교환비율·유효기간은 팀 결정(05번) 반영
- 검증: 정상 생성/404/집계 일치/캐시 적립·사용 잔액 정확

### 7단계 — 소상공인 API 수집·지역 확장
- `소상공인_상가정보_API_명세서.md` 기준. serviceKey 발급 후 `storeListInAdmi`(양덕동 행정동)로 점포 수집
- `clbizDt`(폐업일) 유무로 영업/폐업 분리 → 영업분: 업종구성·경쟁점 / 폐업분: 공실 후보·이전 업종(`prev_industry`)
- 로드뷰·건축물대장으로 공실 확정 → `vacancies`에 실데이터 적재(evidence 기록)
- **지역 확장:** `region_code`만 바꾸면 경북 타 지역 수집되게. 하드코딩 금지
- 인구·유동인구는 이 API에 없음 → 필요 시 SGIS 등 별도(04번 검증표 참조)

## A가 반드시 지킬 것

- 스키마를 임의로 바꾸지 않는다. 컬럼 추가 필요 시 B에게 알리고 공통 문서 갱신 후 반영
- **v2 통지: B 구현 중 계약에 편입된 추가 3건** — `industries.licenses`(인허가), `campaign.is_seed`, `votes.voter_name`(≤30자, 고객 명단용). id는 TEXT형("cafe", "V-A"). AB공통 v2 스키마가 최신
- **마이그레이션 규칙:** 컬럼 추가 시 CREATE IF NOT EXISTS만으로는 기존 DB에 반영 안 됨(voter_name 이슈 실증) → ALTER TABLE 제공 또는 DB 재생성 절차 명시
- B에게 주는 집계 형태 `(vacancy_id, industry_id)→투표수`를 안정적으로 유지 — **held/settled만 집계**(환불·캐시전환 제외)가 계약
- 시드/실데이터 구분(`is_seed`) 철저 — 시연 정직성 요건
- 대량 수집 시 페이징·캐싱 (경북 전체 실시간 로딩 금지)

## A의 결정 대기 (팀·B와 조율)

- 소상공인 serviceKey, 양덕동 행정동 코드 (04번)
- 캐시 모델: 교환비율(1캐시=1원?), 유효기간, 재투표 시 현금 동일취급 여부 (05번)
- 스키마 최종 확정 (B와 합의)
