# 09. 구현 분담 보드 (v2 목표 기준 — 2인 분담 확정)

**2026.07.15.** 동네 투표 전환(08번) 이후 구현해야 할 기능 전체와 A·B 분담. 
분담 원칙은 기존 계약 유지: **A = 데이터가 들어오는 쪽, B = 데이터가 나가는 쪽, 접점 = DB 스키마 하나.**
LLM은 **GPT API(OpenAI, 팀 보유 키)** 확정.

## 담당 A — 데이터·수집 (요약)

**P1이 최우선.** 스키마 v3를 만들어야 B가 P2를 시작할 수 있다.
1. P1: 스키마 v3 + 마이그레이션 + 새 집계 뷰(#1·#4) → B와 계약 합의
2. 투표 v2 + 수요 리스트 API(#2·#3) — 격자 반올림은 서버에서(원좌표 미저장)
3. 6단계 A 몫: cash_ledger·placements(#10·#11)
4. 7단계: 소상공인 수집(#12)·중개사 등록(#13)·주민 측 화면(#14)
- 선행 준비물: serviceKey 발급, 양덕동 행정동 코드 확인(시드 4711158000 검증)

## 담당 B — 엔진·산출 (요약)

**P1 합의 즉시 더미 grid 데이터로 병렬 시작 가능**(순수 함수 원칙 덕).
1. P2: 거리 감쇠 demand(#5) + 헝가리안 재검증(#6) — 검증: 감쇠 단조성·경계값(500/800)·손계산
2. P3: 리포트 v2(#7) — 검증: 지도·리포트 score 동일성, 리스트↔리포트 수치 일치
3. P4: GPT 해설(#8) — 검증: 입력 외 숫자 0건, 금칙어 0건
4. 6단계 B 몫: 상태 전이 규칙(#9) / 7단계: 창업자 측 화면(#15)
- 선행 준비물: OpenAI 키를 .env로(평문 금지), 테스트 골격(97개) v2 재작성 계획

## 병렬 진행 구조 (서로 기다리는 지점 최소화)

```
A: P1 스키마 ──→ 투표 v2·리스트 API ──→ 6단계 A몫 ──→ 7단계 수집·등록·주민 화면
        │(계약 합의)                          ↑(placements 스키마는 P1에서 선생성)
B: 더미 grid로 P2 엔진 ──→ P3 리포트 ──→ P4 GPT ──→ 6단계 B몫 ──→ 창업자 화면
```
- 유일한 동기화 지점 = **P1 스키마 합의**(votes v3·집계 계약·placements). 이후 A·B는 거의 독립.
- placements 테이블은 로직(6단계) 전에 **P1에서 스키마만 선생성** — 마이그레이션 1회로 끝내기 위함.

## 컷라인 (시간 부족 시)

6단계는 화면 흐름만(#9~#11 축소) / #13은 시드 대체 / **#5~#8(수요 엔진→리포트→GPT 해설)과 #6(헝가리안)은 못 뺀다** — AI 활용도 25점의 실체.
