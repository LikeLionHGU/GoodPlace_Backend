## 1. 스키마 변경 (A·B 계약 — 동기화 지점)

```
votes v3:   vacancy_id(전환기 nullable 유지 → 후 제거) + region_code + industry_id + voter_id
            + voter_name + voter_grid(200m 격자) + weight(1/N 다중투표) + amount_won
            + payment_status + created_at + is_seed
vacancies:  + building_use, facilities(JSON), rent_terms, source_tag   (floor 기존)
placements: id, vacancy_id, industry_id, founder_id, campaign_id,
            confirmed_by, confirmed_at, status   (신설)
cash_ledger: reason 에 'topup'(충전) 추가
집계뷰:      vote_grid_counts_view = (industry_id, region_code, voter_grid)→weight 합
            (held/settled만). 공실 수요 = 반경 내 grid 거리 가중 합.
```
**큰 계약 변경 + 마이그레이션 필수.** 전환기에는 v1 뷰/컬럼을 유지(additive)해 회귀를 깨지 않고,
v2 경로가 완성되면 v1 잔재(vacancy_id 등)를 정리한다.

## 2. 단계별 전환 (개발→검증 게이트)

| 단계 | 내용 | 담당 | 검증 게이트 |
|---|---|---|---|
| **P0** | 스키마 v3 additive 마이그레이션 + placements + grid 집계뷰 | A(B 합의) | 마이그레이션 동작·새 뷰 조회·기존 회귀 green |
| **P1** | 냥 캐시(충전·차감·잔액) + 투표 v2(다중·격자·냥·1/N) + 업종 트리 API | A | 다중투표 냥 차감·격자 저장·1/N 집계·잔액부족 |
| **P2** | 거리감쇠 demand(w1 폐기) + floor_fit + grid 어댑터 | B | 감쇠 단조성·경계값(500/800)·floor 계수·손계산·breakdown |
| **P3** | 리포트 v2(업종→공실 1·2·3위+좌표) + /neighborhood + /map | B·A | 리스트↔리포트 일치·지도=리포트 score 동일·좌표 |
| **P4** | placements(성사 모의) + /me/* + 쿠폰 사용처 제한 | A | 성사 상태전이·쿠폰 placements 매장만 |
| **P5** | 로그인/세션(자체) — 소셜 후순위 | A | 세션 발급·voter_id 연결 |
| **P6** | 리포트 화면 문구 확정 반영(구 가중합·'주거형' 제거, floor 표시) | B | 금칙 문구 제거·출처 태그 |

- AI 해설(`ai_explain`): P3에서 v2 리포트 facts만 맞추면 재사용.
- floor_fit: P2 안이지만 v1 엔진에도 얹혀 독립 선작업 가능(작고 승인됨).
