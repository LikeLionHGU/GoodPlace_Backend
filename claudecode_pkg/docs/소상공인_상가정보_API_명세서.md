# 소상공인 상가(상권)정보 API 명세서

**팀 사다리타기 · 2026.07.14 · 동네주문(임시명) 백엔드용**
**출처:** 업로드 3파일 — ① OpenApi 활용가이드(hwp) ② 업종분류(247)·연계표(xlsx) ③ 주요상권현황 CSV(20240101 기준). 제공기관: 소상공인시장진흥공단, 공공데이터포털 서비스 ID `B553077`.

> **표기 규칙:** 본 문서에서 [확인]은 업로드 파일에서 직접 확인한 사실, [표준]은 이 API가 공공데이터포털 공개 표준 서비스라 명칭·구조로 특정되는 사항, [확인필요]는 실제 호출로 검증해야 하는 사항이다. 파라미터명·응답필드·CSV 컬럼·업종분류는 전부 [확인]이다.

## 1. 한눈에 보기

이 API로 **개별 점포(상가업소) 목록**과 **상권 영역 정보**, **업종 분류 코드**를 가져올 수 있다. 우리 서비스에서의 쓰임은 세 가지다.

1. **양덕동 현재 영업 점포 전수 조회** → 어떤 업종이 몇 개 있는지 파악 (적합도 산식의 '주변 업종 구성', '경쟁 밀도' 입력값)
2. **폐업 점포 조회** → 공실 후보 특정의 출발점 (개업일자 `opbizDt`·폐업일자 `clbizDt` 필드 활용)
3. **업종 분류 코드 체계 확보** → 투표 업종 선택지와 점포 데이터의 업종을 같은 코드로 매칭

한 가지 중요한 한계를 먼저 밝힌다. 이 API는 **'영업 중인 점포'를 제공하는 것이 기본**이며, '이 점포는 공실'이라고 직접 알려주는 필드는 없다[표준]. 공실은 폐업 데이터·현재 영업 목록·지도 확인을 조합해 추정해야 한다(4절).

## 3. 오퍼레이션 목록 [확인 — 명칭 / 표준 — 기능]

활용가이드에서 확인된 오퍼레이션은 다음과 같다. 명칭은 [확인], 각 기능 설명은 명칭과 공개 표준에 근거한 [표준]이며 필수 파라미터는 첫 호출로 [확인필요].

### 3-1. 점포(상가업소) 조회 — 우리 핵심

| 오퍼레이션 | 무엇을 주나 | 주요 파라미터 |

| `storeListInRectangle` | 사각형(경위도 min/max) 안의 점포 목록 | `minx`, `miny`, `maxx`, `maxy` |
| `storeListInAdmi` | 행정동 코드 기준 점포 목록 | `divId=adongCd`, `key=행정동코드` |
| `storeListInArea` | 지정 영역(행정구역 등) 점포 목록 | 지역 코드 계열 |
| `storeListInDong` | 법정동 기준 점포 목록 | `divId=ldongCd`, `key=법정동코드` |
| `storeListInPnu` | 필지(PNU) 기준 점포 목록 | `key=PNU코드` |
| `storeListInBuilding` | 건물 기준 점포 목록 | 건물관리번호 `bldMngNo` |
| `storeListInUpjong` | 업종코드 기준 점포 목록 | `indsLclsCd`/`indsMclsCd`/`indsSclsCd` |
| `storeListByDate` | 등록/수정일 기준 점포 목록 | `stdrYm` 등 |
| `storeOne` | 단일 점포 상세 (점포ID) | `key=bizesId` |
| `reqStoreModify` | 점포정보 수정 요청 | (등록/정정용) |

> 양덕동 개발에는 **`storeListInAdmi`(행정동=양덕동 코드) 또는 `storeListInRadius`(양덕동 중심 반경)**가 1순위다. 행정동 코드는 CSV·행정표준코드로 확보한다.

| 오퍼레이션 | 무엇을 주나 |

상권 영역은 §5의 CSV(주요상권현황)와 같은 성격이며, 상권 폴리곤 좌표(`coords`)와 면적(`trarArea`)을 준다.

## 4. 점포 응답 필드 [확인] — 공실 특정의 핵심

`storeList*` 계열이 반환하는 점포 레코드의 필드다. 활용가이드에서 확인된 필드 전체:

**식별·명칭**
`bizesId`(상가업소번호), `bizesNm`(상호명), `brchNm`(지점명)

**업종 분류** (§6 코드와 연결)
`indsLclsCd`/`indsLclsNm`(대분류), `indsMclsCd`/`indsMclsNm`(중분류), `indsSclsCd`/`indsSclsNm`(소분류), `ksicCd`/`ksicNm`(표준산업분류 10차)

**행정구역**
`ctprvnCd`/`ctprvnNm`(시도), `signguCd`/`signguNm`(시군구), `adongCd`/`adongNm`(행정동), `ldongCd`/`ldongNm`(법정동)

**주소·위치**
`rdnmAdr`(도로명주소), `lnoAdr`(지번주소), `bldNm`(건물명), `dongNo`(동번호), `flrNo`(층번호), `hoNo`(호번호), `bldMngNo`(건물관리번호), `rdnmCd`(도로명코드), `lnoCd`(지번코드), `plotSctCd`/`plotSctNm`(대지구분), `lnoMnno`/`lnoSlno`(지번 본/부번), `bldMnno`/`bldSlno`(건물 본/부번), `oldZipcd`/`newZipcd`(우편번호), **`lon`(경도), `lat`(위도)**

**영업 이력** ← 공실 추정에 직결 [확인]
`opbizDt`(개업일자), `clbizDt`(폐업일자), `chgGb`(변경구분), `chgDt`(변경일자), `stdrDt`/`stdrYm`(데이터 기준일)

### 공실 특정에 이 필드를 어떻게 쓰나 (답변집 부록 E 파이프라인 정련)

1. **양덕동 현재 영업 점포 확보:** `storeListInAdmi`로 양덕동 행정동 코드를 넣어 `clbizDt`(폐업일자)가 비어 있는 = 영업 중 점포를 전부 받는다. → 업종별 개수 집계 = 산식의 경쟁 밀도·업종 구성 입력.
2. **폐업 이력 확보:** 같은 조회에서 `clbizDt`가 채워진 레코드, 또는 `storeListByDate`로 최근 폐업분을 받는다. → 폐업 점포의 주소·좌표·직전 업종(`indsSclsNm`)이 **공실 후보 + '이전 매장이 무엇이었나'** 정보가 된다.
3. **공실 확정:** 폐업 주소에 현재 영업 점포가 없는지 1번 목록과 대조 → 카카오/네이버 로드뷰로 육안 확인 → 건축물대장으로 면적·층 보강.

> 주의: 이 API의 폐업 데이터 갱신 주기·과거 이력 보존 범위는 [확인필요]다. 폐업 후 신규 점포가 같은 주소에 들어왔으면 공실이 아니므로 대조는 필수다. `opbizDt`/`clbizDt`가 어느 시점까지 소급되는지도 첫 호출에서 확인한다.

## 5. 주요상권현황 CSV [확인]

업로드된 `소상공인시장진흥공단_주요상권현황_20240101.csv`의 실측 구조다.

- **인코딩:** CP949(EUC-KR 계열). UTF-8로 읽으면 깨진다. 파싱 시 `encoding='cp949'`, 그리고 상권좌표 필드가 길어 `csv.field_size_limit` 상향 필요.
- **행수:** 1,227개 상권 (전부 유형구분 '주요상권')
- **컬럼:** `상권번호`, `상권명`, `유형구분`, `시도코드`, `시도명`, `시군구코드`, `시군구명`, `상권좌표수`, `상권좌표`(경위도 `lon,lat` 쌍을 `|`로 연결한 폴리곤), `데이터기준일자`
- **포항 상권:** 25개 확인 (예: 11081 효곡동행정복지센터, 11084 포항시외터미널, 11086 상대동행정복지센터 등). **'양덕' 명칭 상권은 이 CSV에 없음** → 양덕동은 이 '주요상권' 목록 밖이므로, 양덕동 점포는 상권 단위가 아니라 **행정동/반경 기준 점포 조회(`storeListInAdmi`/`storeListInRadius`)로 접근**해야 한다.

CSV는 상권 폴리곤을 미리 확보해두는 용도이고, 실시간 점포 데이터는 §3 API로 가져온다.

## 6. 업종분류표 xlsx [확인]

업로드된 `업종분류(2302) 및 연계표 v1.xlsx`는 시트 3개다.

1. **상권_업종분류(247):** 현행 247개 업종 코드. 컬럼 `대분류코드/명`(예: G2 소매업), `중분류코드/명`(G204 종합소매업), `소분류코드/명`(G20405 편의점). → 투표 업종 선택지와 점포 데이터 매칭의 기준표. 우리 시드 업종 6종(카페·베이커리 등)을 이 코드에 매핑해야 데이터가 붙는다.
2. **상권_업종연계표(837→247):** 구 분류(8xx, 예 D23A04 타이어판매) ↔ 신 분류(247) 대응표. 과거 데이터를 다룰 때 코드 변환용.
3. **표준산업분류(10차) 연계표:** 247 분류 ↔ 통계청 KSIC 10차(`ksicCd`) 대응. 점포 응답의 `ksicCd`와 인구·통계 데이터를 연결할 때 쓴다.

## 7. 호출 예시 (구성 방식) [표준 — serviceKey·코드는 실제 값으로 교체]

```
# 양덕동(행정동코드 예: 47111xxxxx) 점포 목록 — JSON, 100건
http://apis.data.go.kr/B553077/api/open/sdsc2/storeListInAdmi
  ?serviceKey={인증키}
  &divId=adongCd
  &key={양덕동_행정동코드}
  &numOfRows=100
  &pageNo=1
  &type=json

# 반경 조회 — 양덕동 중심 좌표 반경 500m
http://apis.data.go.kr/B553077/api/open/sdsc2/storeListInRadius
  ?serviceKey={인증키}
  &radius=500
  &cx={경도}&cy={위도}
  &type=json

# 업종 대분류 목록
http://apis.data.go.kr/B553077/api/open/sdsc2/largeUpjongList
  ?serviceKey={인증키}&type=json
```

> 파라미터명(`divId`, `key`, `radius`, `cx`/`cy`)은 [확인]이나, 각 오퍼레이션의 **필수/선택 구분과 좌표 파라미터 정확 명칭(cx/cy vs lon/lat)**은 첫 호출로 [확인필요]. 공공데이터포털 서비스 페이지의 '미리보기'로 1건 테스트가 가장 빠르다.

## 8. 백엔드 연동 시 확정해야 할 것 (개발계획서 §4 연계)

- [ ] 공공데이터포털에서 `B553077` 서비스 활용신청 → serviceKey 발급 (일반 인증키/Decoding 키 구분 확인)
- [ ] 양덕동 **행정동 코드** 확정 (행정표준코드관리시스템 또는 CSV 시군구코드에서)
- [ ] `storeListInAdmi` 1건 호출로 응답 필드·좌표 파라미터명·`clbizDt` 채움 여부 [확인필요] 검증
- [ ] 시드 업종 6종 ↔ 247 업종코드 매핑표 작성 (§6 시트1 기준)
- [ ] 폐업 데이터 소급 범위·갱신 주기 [확인필요] → 공실 추정 신뢰도 판단
- [ ] 일일 트래픽/호출 한도 [확인필요] → 양덕동 전수 수집 시 페이징 계획

## 9. 요약

양덕동 개발 기준으로, 이 API의 실전 경로는 **`storeListInAdmi`(양덕동 행정동)로 점포 전수 수집 → `clbizDt` 유무로 영업/폐업 분리 → 영업분으로 업종 구성·경쟁 밀도 계산, 폐업분으로 공실 후보·이전 업종 확보 → 로드뷰·건축물대장으로 공실 확정**이다. 상권 CSV에 양덕동이 없으므로 상권 단위가 아닌 행정동·반경 조회가 정답이고, 업종은 반드시 247 코드 체계로 통일해 투표·점포·산식이 같은 언어를 쓰게 한다. serviceKey 발급과 첫 호출 검증이 다음 실행 항목이다.
